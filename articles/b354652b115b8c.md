## はじめに

「Macの蓋を閉じたら、自動で何か処理を走らせたい」——そんなこと思ったことはありませんか？

例えば：
- 退勤時に蓋を閉じたら、勤怠を自動打刻したい
- スリープしたらSlackのステータスを「離席中」に変えたい
- 作業ログを自動で記録したい

今回は **n8n**（ノーコード自動化ツール）と **sleepwatcher**（macOS のスリープ検知ツール）を組み合わせて、Mac を閉じた瞬間にワークフローが発火する仕組みを作ってみました。

## 全体構成

```
Mac スリープ検知（sleepwatcher）
    ↓ curl で HTTP POST
n8n Webhook Trigger
    ↓
任意の処理（Slack通知、ログ記録、API呼び出しなど）
```

n8n 単体では macOS のシステムイベントを直接検知できません。そこで、macOS 側でスリープを検知し、n8n の Webhook エンドポイントに HTTP リクエストを送る構成にしています。

## 環境

- macOS（Apple Silicon / Intel どちらでも可）
- n8n（ローカルで動作させる前提）
- Homebrew

## 手順

### 1. n8n のセットアップ

n8n をまだインストールしていない場合：

```bash
# npm でグローバルインストール
npm install -g n8n

# 起動
n8n start
```

Docker で動かす場合：

```bash
docker run -it --rm \
  --name n8n \
  -p 5678:5678 \
  -v n8n_data:/home/node/.n8n \
  docker.n8n.io/n8nio/n8n
```

起動後、`http://localhost:5678` にアクセスできることを確認します。

### 2. n8n でワークフローを作成

#### Webhook トリガーの設定

1. n8n の UI で「New Workflow」を作成
2. 「Webhook」ノードを追加
3. 以下のように設定：

| 設定項目 | 値 |
|---------|-----|
| HTTP Method | POST |
| Path | `mac-sleep` |

これで `http://localhost:5678/webhook/mac-sleep` というエンドポイントが生成されます。

#### 後続ノードの追加（例：Slack通知）

Webhook ノードの後ろに好きな処理を繋げます。例えば Slack に通知を送る場合：

1. 「Slack」ノードを追加して Webhook ノードに繋ぐ
2. チャンネルとメッセージを設定：

```
💤 {{$json["hostname"]}} がスリープしました（{{$json["timestamp"]}}）
```

**ワークフローを「Active」にするのを忘れずに！**

### 3. sleepwatcher のインストール

macOS のスリープ/復帰イベントを検知できるツール `sleepwatcher` を Homebrew でインストールします。

```bash
brew install sleepwatcher
```

サービスとして常駐起動させます：

```bash
brew services start sleepwatcher
```

### 4. スリープ時スクリプトの作成

sleepwatcher は、スリープ時に `~/.sleep` ファイルを自動実行します。

```bash
cat << 'EOF' > ~/.sleep
#!/bin/bash
curl -s -X POST http://localhost:5678/webhook/mac-sleep \
  -H "Content-Type: application/json" \
  -d "{\"event\": \"sleep\", \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\", \"hostname\": \"$(hostname)\"}"
EOF

chmod +x ~/.sleep
```

これだけで、Mac を閉じるたびに n8n の Webhook が叩かれるようになります。

### 5.（おまけ）復帰時のイベントも捕捉する

蓋を開けた時（スリープ復帰時）にも処理を走らせたい場合は `~/.wakeup` を作成します。

```bash
cat << 'EOF' > ~/.wakeup
#!/bin/bash
curl -s -X POST http://localhost:5678/webhook/mac-wakeup \
  -H "Content-Type: application/json" \
  -d "{\"event\": \"wakeup\", \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\", \"hostname\": \"$(hostname)\"}"
EOF

chmod +x ~/.wakeup
```

n8n 側にも `mac-wakeup` パスの Webhook トリガーを追加すれば、開閉どちらのイベントも拾えます。

## 動作確認

### 手動テスト

まずは手動で curl を叩いて、n8n 側が正しく反応するか確認しましょう。

```bash
curl -X POST http://localhost:5678/webhook/mac-sleep \
  -H "Content-Type: application/json" \
  -d '{"event": "sleep", "timestamp": "2026-03-17T10:00:00Z", "hostname": "my-mac"}'
```

n8n の実行履歴に記録が残っていれば OK です。

### 実際にスリープさせてテスト

```bash
# Mac を即座にスリープさせるコマンド
pmset sleepnow
```

スリープ後に n8n の実行履歴を確認して、ワークフローが発火していれば成功です 🎉

## 注意点・ハマりどころ

### スリープ前にリクエストが完了するか？

スリープ直前に curl が実行されるため、**リクエスト送信完了前に Mac がスリープしてしまう**可能性があります。ローカルの n8n であればほぼ問題ありませんが、リモートの n8n（クラウド版など）を使う場合はネットワーク切断が先に起こるリスクがあります。

対策としては、`~/.sleep` を以下のように修正します：

```bash
#!/bin/bash
# リクエスト完了まで最大5秒待つ
curl -s --max-time 5 -X POST http://localhost:5678/webhook/mac-sleep \
  -H "Content-Type: application/json" \
  -d "{\"event\": \"sleep\", \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\", \"hostname\": \"$(hostname)\"}"
```

### sleepwatcher が動いているか確認する

```bash
brew services list | grep sleepwatcher
```

`started` と表示されていれば OK です。

### スクリプトに実行権限があるか

```bash
ls -la ~/.sleep ~/.wakeup
```

`-rwx` になっていなければ `chmod +x` を忘れています。

## 代替手段：Hammerspoon を使う方法

sleepwatcher の代わりに [Hammerspoon](https://www.hammerspoon.org/) を使う方法もあります。Hammerspoon は Lua スクリプトで macOS を自動化できるツールです。

```lua
-- ~/.hammerspoon/init.lua
local sleepWatcher = hs.caffeinate.watcher.new(function(event)
  if event == hs.caffeinate.watcher.systemWillSleep then
    hs.http.asyncPost(
      "http://localhost:5678/webhook/mac-sleep",
      hs.json.encode({
        event = "sleep",
        timestamp = os.date("!%Y-%m-%dT%H:%M:%SZ"),
        hostname = hs.host.localizedName()
      }),
      { ["Content-Type"] = "application/json" },
      function(status, body, headers)
        -- コールバック（ログ出力など）
      end
    )
  elseif event == hs.caffeinate.watcher.systemDidWake then
    hs.http.asyncPost(
      "http://localhost:5678/webhook/mac-wakeup",
      hs.json.encode({
        event = "wakeup",
        timestamp = os.date("!%Y-%m-%dT%H:%M:%SZ"),
        hostname = hs.host.localizedName()
      }),
      { ["Content-Type"] = "application/json" },
      function(status, body, headers) end
    )
  end
end)
sleepWatcher:start()
```

Hammerspoon の方が高機能ですが、sleepwatcher の方がシンプルで導入が楽です。用途に応じて選んでください。

## 応用アイデア

この仕組みをベースに、いろいろな自動化ができます：

| ユースケース | 実装 |
|-------------|------|
| 退勤打刻 | スリープ時に勤怠管理 API を叩く |
| Slack ステータス変更 | スリープ→離席、復帰→オンラインに自動切替 |
| 日報の自動生成 | 復帰時にその日の作業ログをまとめて Slack に投稿 |
| スマートホーム連携 | 蓋を閉じたらデスクライトを消灯（Hue API 連携） |
| セキュリティ通知 | 深夜にスリープ復帰したら Slack にアラート |

## クラウド版 n8n を使う場合（n8n Cloudなど）

この記事ではローカル環境の n8n を前提に解説しましたが、n8n の公式クラウドサービスである [n8n Cloud](https://n8n.cloud/) や、自前でサーバーにホストした n8n を使うこともできます。

クラウド版を利用する場合、主に以下の2点がローカル版と異なります。

### 1. Webhook URL の変更

Webhook の URL が `localhost` ではなく、お使いの n8n インスタンスの URL になります。n8n Cloud の場合は、以下のような形式です。

```
https://YOUR_SUBDOMAIN.n8n.cloud/webhook/mac-sleep
```

### 2. 認証の設定

クラウド版の Webhook は、セキュリティのために認証が必要になることが一般的です。n8n Cloud の場合、Webhook ノードで認証方法として `Header Auth` を選択し、APIキー（Header Name: `X-N8N-API-KEY`）を設定します。

### スクリプトの修正

上記を踏まえて、`~/.sleep` スクリプトを以下のように修正します。

```bash
#!/bin/bash

# ご自身の n8n Cloud の URL と APIキーに置き換えてください
N8N_WEBHOOK_URL="https://YOUR_SUBDOMAIN.n8n.cloud/webhook/mac-sleep"
N8N_API_KEY="YOUR_N8N_API_KEY"

# リクエスト完了まで最大10秒待つ
curl -s --max-time 10 -X POST "$N8N_WEBHOOK_URL" \
  -H "Content-Type: application/json" \
  -H "X-N8N-API-KEY: $N8N_API_KEY" \
  -d "{\"event\": \"sleep\", \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\", \"hostname\": \"$(hostname)\"}"
```

**ポイント：**
- **URLとAPIキーを変数に**：管理しやすくなります。
- **`--max-time` を長めに設定**：クラウドとの通信はネットワーク遅延があるため、タイムアウト時間をローカルの場合より少し長め（10秒など）に設定すると、スリープ前にリクエストが失敗するリスクを減らせます。

これで、Mac を閉じると安全にクラウド上の n8n ワークフローを呼び出すことができます。

## まとめ

- **sleepwatcher** で macOS のスリープイベントを検知
- **curl** で n8n の Webhook エンドポイントに POST
- n8n 側で好きなワークフローを組む

たった 3 ステップで、Mac の蓋の開閉をトリガーにした自動化が実現できます。n8n のノーコード UI で後続処理を自由に組み替えられるのが便利ですね。
